# Архитектура финального решения

## Общая схема

Сервис решает задачу онлайн-рекомендации работников для смен. Внешняя система отправляет данные и вызывает RPC-методы, а сервис хранит историю, готовит модель и возвращает список кандидатов.

Ключевой принцип: тяжелая работа делается в `prepare`, быстрый ответ делается в `predict`.

## Основные компоненты

| Компонент | Назначение |
|---|---|
| `hackaton/service/app.py` | RPC-методы сервиса |
| `hackaton/service/repositories.py` | запись и чтение данных из SQLite |
| `hackaton/service/prepare_manager.py` | подготовка модели и кэшей |
| `hackaton/service/ml_model.py` | ML-признаки, обучение, скоринг и rerank |
| `hackaton/eval/evaluator.py` | локальная симуляция проверки |
| `hackaton/eval/metric.py` | расчет целевой метрики |
| `hackaton/train/cv.py` | time-based cross-validation без запуска сервиса |

## RPC-методы

Сервис поддерживает основные методы:

- `user` - загрузить или обновить пользователей;
- `shift` - загрузить или обновить смены;
- `event` - загрузить события;
- `prepare` - подготовить модель и кэши;
- `ready` - проверить готовность;
- `predict` - получить кандидатов для смены;
- `health` - проверить, что сервис жив;
- `metrics` - получить технические метрики Prometheus.

## Хранилище

SQLite используется как простое локальное хранилище:

- таблица `users`;
- таблица `shifts`;
- таблица `events`.

Данные попадают туда через RPC-методы `user`, `shift`, `event`.

SQLite нужен для накопления данных и подготовки модели. В нормальном режиме `predict` не должен делать тяжелых запросов к базе.

## `PrepareManager`

`PrepareManager` отвечает за подготовку:

1. Загружает данные из SQLite;
2. Приводит даты к нужным типам;
3. Строит кэш пользователей;
4. Обучает `MLModel`;
5. Строит inference cache;
6. Переводит сервис в состояние готовности.

Внутри хранятся:

- `_users_cache` - данные пользователей по `user_id`;
- `_location_cache` - пользователи по локациям;
- `_global_top` - глобальный список активных пользователей;
- `model` - обученная ML-модель.

## `MLModel`

`MLModel` делает три вещи:

1. Строит статистики пользователей и работодателей;
2. Формирует обучающую выборку;
3. Скорит кандидатов для новой смены.

Финальная модель - `LGBMClassifier`.

Если LightGBM недоступен, в коде есть fallback на `LogisticRegression`, но основной ожидаемый путь - LightGBM.

## Candidate cache

Для каждой локации заранее строится список пользователей, отсортированных по активности.

При `predict`:

- если в локации достаточно пользователей, берутся только они;
- если пользователей мало, список дополняется глобальным топом;
- дальше берется ограниченный пул до 300 кандидатов.

Такой подход снижает шум и ускоряет inference.

## Inference cache

После обучения модель строит numpy-массивы с пользовательскими признаками.

Это нужно, чтобы при `predict` не пересчитывать все с нуля для каждого кандидата. Сервис быстро собирает матрицу признаков для текущей смены и пачки кандидатов, затем одним вызовом получает score.

## Rerank

После ML-score применяется дополнительная сортировка:

1. Пользователи, которые уже подавали заявку на эту конкретную смену.
2. "Спящие" пользователи: есть `FINISHED`, но нет `APPLY` в обучающем окне.
3. Остальные пользователи по ML-score.

Это помогает учесть паттерны, которые модель видит неидеально из-за ограниченного исторического окна.

## Поток данных в eval

Eval работает как дневная симуляция:

1. Загружает train-данные в сервис;
2. Вызывает `prepare`;
3. Для каждого validation-дня вызывает `predict` по сменам;
4. Сравнивает top-кандидатов с `apply.csv`;
5. Считает метрику;
6. После дня догружает validation-события и вызывает инкрементальный `prepare`.

Так проверяется не только качество модели, но и то, что сервис выдерживает реальный контракт.

## Почему не SQL в `predict`

SQL-запросы в `predict` плохо подходят для "горячего" пути:

- запросы могут быть медленными;
- сложные признаки тяжело собирать на лету;
- latency становится менее стабильной;
- модельные признаки должны быть согласованы с обучением.

Поэтому финальная архитектура переносит тяжелую работу в `prepare`, а `predict` оставляет быстрым и предсказуемым.
