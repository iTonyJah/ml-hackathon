# Метрики и эксперименты

## Целевая метрика

Решение оценивается как ранжирование кандидатов для смен.

Для каждой смены сервис возвращает список пользователей. Eval смотрит TOP-10 и сравнивает его с фактическими заявками из `apply.csv`.

Используется ROC-AUC с ограничением FPR:

```text
max_fpr = min(1.0, capacity / 10)
```

Агрегация:

1. Метрика по смене.
2. Среднее по группам `capacity` внутри дня.
3. Среднее по дням.
4. Итоговая метрика.

## Почему важно смотреть на `capacity`

`capacity` показывает, сколько людей нужно на смену.

Если `capacity = 1`, оценка может стать неинформативной: в верхней части списка всего один кандидат, а ROC-AUC требует сравнения разных классов. Поэтому часть смен может не войти в оценку. Если в дне все полезные заявки относятся только к таким сменам, дневная метрика может быть нулевой не из-за плохой модели, а из-за структуры данных.

## Основные эксперименты

### Baseline

Изначально сервис ранжировал кандидатов очень просто: локация, наличие медкнижки и почти случайный порядок по `id`.

Такой baseline дал ориентировочную метрику около `0.534`.

### LightGBM

Затем была добавлена модель `LGBMClassifier`.

Она использует:

- историю пользователя;
- параметры смены;
- совместимость пользователя и смены;
- историю по работодателю и рабочей точке;
- признаки повторяющихся смен.

### Candidate limit

Сначала идея была брать большой пул кандидатов, например 5000 пользователей, и отдавать их модели.

На практике это добавило много шума. Пользователи из нерелевантных локаций могли мешать правильному ранжированию.

Пул сократили до 300 кандидатов. Это ускорило inference и улучшило качество.

### Sleeper-rerank

Была найдена группа "спящих" работников:

- `APPLY = 0`;
- `FINISHED > 0`.

Скорее всего, их заявки были до начала обучающего окна, но выходы на смены попали в историю. Для модели они выглядят слабыми, но на самом деле это надежные работники.

Для них добавили отдельный tier в post-processing.

### AUC early stopping

Для `LGBMClassifier` добавили early stopping по AUC.

Идея простая: обучаем много деревьев, но останавливаемся, когда качество на отложенной части выборки перестает улучшаться.

Это лучше соответствует финальной метрике, чем обычный logloss.

### LGBMRanker

Также проверялся `LGBMRanker` с LambdaRank.

Идея была логичной: задача ведь про ранжирование. Но на этих данных ranker оказался хуже классификатора.

Основные причины:

- группы по сменам часто маленькие;
- негативные примеры получались слишком узкими;
- без надежного early stopping ranker был менее стабилен;
- официальный validation слишком мал, чтобы уверенно сравнивать такие модели.

## Time-based CV

Официальный validation может быть маленьким и шумным. Поэтому был добавлен time-based CV.

Принцип:

1. Берем исторические события.
2. Отрезаем последние N дней как validation.
3. Обучаемся только на прошлом.
4. Проверяемся на будущем.

Это ближе к реальному режиму работы сервиса.

Результаты из рабочих отчетов:

| Модель | CV-метрика | Оцениваемые смены |
|---|---:|---:|
| `LGBMClassifier` | 0.7612 | 121 |
| `LGBMRanker` | 0.6661 | 137 |

Вывод: оставлен `LGBMClassifier`.

## Сводная таблица результатов

| Этап | Метрика | Контекст |
|---|---:|---|
| Baseline | 0.534 | стартовое SQL-ранжирование |
| ML + candidate cache | около 0.617 | первый ML-подход |
| Candidate limit + sleeper-rerank | около 0.667 | улучшение пула и post-processing |
| Локальный validation после фиксов | 0.800 | исправленный split и target |
| Time-based CV, Classifier | 0.7612 | более надежная внутренняя оценка |
| Новый увеличенный датасет | 0.978 | больше данных и полноценные оцениваемые дни |

## Как читать эти числа

Важно не сравнивать все значения как одну leaderboard-линию.

Разные числа получены на разных наборах и в разных условиях:

- часть результатов относится к старому validation;
- часть - к исправленному split;
- часть - к time-based CV;
- часть - к новому увеличенному датасету.

Поэтому рядом с каждой метрикой нужно всегда указывать:

- какие данные использовались;
- как был сделан split;
- сколько было оцениваемых дней;
- сколько было оцениваемых смен.

## Главный вывод

Финальный выбор - `LGBMClassifier` с AUC early stopping, candidate limit 300 и sleeper-rerank.

Это решение оказалось лучше по балансу:

- качество;
- скорость;
- стабильность;
- простота поддержки;
- соответствие online-сервису.
