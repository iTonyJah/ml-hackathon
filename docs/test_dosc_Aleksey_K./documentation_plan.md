# План подготовки итоговой документации проекта

## Ограничения работы

- Все новые материалы сохраняются только в этой папке: `docs/test_dosc_Aleksey_K.`
- Существующие файлы проекта не редактируются.
- Существующие файлы проекта не удаляются.
- Если для финализации нужно изменить уже существующий документ, это фиксируется в `edit_docs.md`.

## Цель

Подготовить итоговую документацию по проекту так, чтобы ее мог понять человек без глубокого ML-бэкграунда, но чтобы техническая часть воспроизводимости была оформлена строго и привычно для инженерного сообщества.

Главная идея: не заставлять читателя читать все рабочие отчеты подряд. Нужно сделать один понятный основной маршрут:

1. Что была за задача.
2. Как устроено финальное решение.
3. Почему выбраны именно такие алгоритмы и эвристики.
4. Какие результаты получены.
5. Как полностью воспроизвести результат локально.

## Что уже есть в проекте

Ветка: `release/sleeper-rerank`.

Основные источники для итоговой документации:

- `docs/support/FULL_REPORT.md` - самый полный рабочий отчет.
- `docs/support/REPORT.md` - короткий отчет об улучшении модели.
- `docs/support/CV_REPORT.md` - отчет по time-based cross-validation и сравнению моделей.
- `docs/support/eval_command_analysis.md` - подробный разбор eval-команды.
- `docs/HOW-TO-Veterok.md` - заготовка инструкции по воспроизводимости.
- `README.md`, `HOW-TO.md`, `TRAIN.md`, `DATA.md`, `REGLAMENT.md` - базовые документы проекта.

## Предлагаемый состав итогового пакета документации

### 1. `FINAL_REPORT.md`

Основной итоговый отчет. Пишется простым языком.

Рекомендуемая структура:

1. Краткое резюме проекта.
2. Постановка задачи.
3. Какие данные используются.
4. Как считается качество.
5. Что было в исходном baseline.
6. Как устроено финальное решение.
7. Что делает `prepare`.
8. Что делает `predict`.
9. Какие признаки использует модель.
10. Что такое sleeper-rerank.
11. Как проводились эксперименты.
12. Почему оставлен `LGBMClassifier`, а не `LGBMRanker`.
13. Итоговые результаты.
14. Ограничения решения.
15. Что можно улучшить дальше.

Тон: объяснительный, спокойный, без лишнего академического языка.

### 2. `REPRODUCIBILITY.md`

Строгая инструкция по воспроизводимости.

Здесь стиль должен быть инженерным:

- точные версии и окружение;
- структура входных данных;
- команды установки;
- команды подготовки validation split;
- команды миграции БД;
- команды запуска сервиса;
- команды запуска eval;
- команды запуска CV;
- ожидаемые артефакты;
- типовые ошибки и диагностика.

Важно: команды должны соответствовать текущей структуре проекта. В частности, актуальные split-скрипты лежат в `scripts/`:

```bash
poetry run python scripts/create_validation_split.py
poetry run python scripts/create_validation_split_2d.py
```

### 3. `ARCHITECTURE_OVERVIEW.md`

Короткое описание архитектуры решения.

Рекомендуемая структура:

1. RPC-сервис и методы.
2. SQLite как хранилище входных данных.
3. `PrepareManager` как компонент подготовки.
4. `MLModel` как компонент обучения и скоринга.
5. In-memory cache для быстрого inference.
6. Поток данных в eval.
7. Почему `predict` не должен делать тяжелые SQL-запросы.

Этот документ можно сделать небольшим, чтобы он был удобной картой проекта.

### 4. `METRICS_AND_EXPERIMENTS.md`

Документ про метрики и эксперименты.

Сюда стоит вынести:

- формулу целевой метрики;
- почему `capacity=1` может не давать оцениваемых смен;
- результаты официального eval;
- результаты time-based CV;
- сравнение Classifier и Ranker;
- вывод, почему финальный выбор - `LGBMClassifier`.

### 5. `edit_docs.md`

Список рекомендуемых правок в существующих документах проекта. Этот файл нужен, потому что редактировать текущие документы запрещено.

## Порядок работы

### Этап 1. Зафиксировать факты

Нужно собрать и согласовать одну версию фактов:

- финальная ветка;
- финальная модель;
- набор признаков;
- candidate limit;
- правила sleeper-rerank;
- актуальные команды запуска;
- актуальные результаты eval и CV;
- какие результаты относятся к старому датасету, а какие к новому.

Результат этапа: короткая таблица фактов внутри черновика `FINAL_REPORT.md` или отдельный блок "Зафиксированные факты".

### Этап 2. Собрать основной отчет

Собрать `FINAL_REPORT.md` из уже существующих рабочих отчетов, но переписать его как единый рассказ.

Что важно:

- убрать дубли;
- выровнять нумерацию;
- не смешивать старые и новые метрики без пояснения;
- каждый результат подписывать контекстом данных;
- не перегружать читателя кодом;
- код оставлять только там, где он помогает понять идею.

### Этап 3. Подготовить воспроизводимость

Собрать `REPRODUCIBILITY.md`.

Минимально необходимые сценарии:

1. Установка зависимостей.
2. Подготовка данных.
3. Создание validation split.
4. Миграция БД.
5. Запуск сервиса.
6. Запуск eval.
7. Запуск CV.
8. Проверка тестов.

Отдельно нужно указать, что eval требует уже запущенный сервис.

### Этап 4. Описать архитектуру

Собрать `ARCHITECTURE_OVERVIEW.md` простым языком.

Главная мысль: тяжелая подготовка делается в `prepare`, быстрый ответ делается в `predict`.

### Этап 5. Описать эксперименты

Собрать `METRICS_AND_EXPERIMENTS.md`.

Важно отделить:

- официальный eval;
- локальный validation split;
- time-based CV;
- новый датасет;
- старые промежуточные результаты.

### Этап 6. Сформировать список правок существующих документов

Все нужные изменения в текущих документах записать в `edit_docs.md`.

Сами документы не менять.

### Этап 7. Финальная проверка

Проверить:

- нет ли противоречий по метрикам;
- нет ли устаревших путей команд;
- все ли команды используют актуальные файлы;
- понятно ли, какой документ читать первым;
- воспроизводимость написана достаточно строго;
- остальные разделы написаны простым языком.

## Definition of Done

Документация готова, если:

- есть основной итоговый отчет;
- есть строгая инструкция воспроизводимости;
- есть короткая архитектурная карта;
- есть отдельное описание метрик и экспериментов;
- есть `edit_docs.md` с правками существующих документов;
- читатель может понять проект без чтения всех рабочих отчетов;
- инженер может повторить запуск по командам из `REPRODUCIBILITY.md`;
- старые рабочие отчеты остаются как приложения, а не как основной путь чтения.
