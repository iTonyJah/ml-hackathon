**Дата:** 2026-05-07
**Ветка:** `improve/sleeper-rerank`

## 1. Мотивация

Официальный val-набор имеет критический изъян как инструмент оценки: 12 дней, из которых 3 структурно не оцениваются (capacity=1 смены), итого **11 оцениваемых смен**. При таком размере выборки метрика нестабильна — одна смена меняет итог на 0.09 пункта.

Официальный результат **0.750** не отражает реального качества модели:
- Это потолок именно этого val-набора (3 из 12 дней дают 0.0 независимо от качества модели)
- 9 из 9 оцениваемых дней дают 1.0 — на 11 сменах это не гарантия обобщения

При сдаче работы будет использован **другой** тест-набор. Нужна более надёжная оценка.

## 2. Методология Time-based CV

### 2.1 Принцип

Используем тот же принцип, что и продакшн-цикл: обучаем на исторических данных, оцениваем на последующем периоде.

```
Все тренировочные данные
│
├── Train: события до cutoff
│   └── обучаем MLModel
│
└── Val: события после cutoff (30 дней)
    └── оцениваем по тем же правилам, что и официальный eval
```

**Cutoff:** `max(events.ts) - 30 дней` = 2026-02-14 - 30 дней = **2026-01-15**

### 2.2 Реализация

Новый модуль `hackaton/train/cv.py` воспроизводит продакшн-pipeline без запуска RPC-сервиса:

1. Загружает данные, делит по времени
2. Обучает `MLModel` на pre-cutoff событиях
3. Строит кэш кандидатов из `PrepareManager._build_cache` (те же правила: location_cache + global fallback)
4. Для каждой val-смены с APPLY-событиями: предсказывает top-10, сравнивает с ground truth
5. Применяет ту же функцию `calculate_target_metric` из `hackaton/eval/metric.py`

Запуск:
```bash
poetry run python -m hackaton.train.cli cv \
  --user-path data/train/user.csv \
  --shift-path data/train/shift_train.csv \
  --event-path data/train/event_train.csv \
  --output-dir artifacts/cv_30d \
  --val-days 30
```

### 2.3 Масштаб оценки

| | Официальный eval | Time-based CV (30d) |
|---|---|---|
| Дней в оценке | 12 (9 оцениваемых) | 22 |
| Оцениваемых смен | 11 | 121–137 |
| Shifts с applies | 45 | 6 325 |
| Надёжность | Низкая | Высокая |

CV даёт **в 12 раз больше** данных для оценки.

| evaluated_days | 19 |

### 3.2 LGBMRanker — lambdarank (Шаг 2)

**Идея:** переформулировать задачу как Learning to Rank.

**Данные обучения:**
- Группа = смена (query)
- Позитивы = пользователи с APPLY на эту смену
- Негативы = пользователи с VIEW на ту же смену (без APPLY)
- Ограничение: max 10×positives негативов на группу
- 6 969 групп, 60 895 обучающих примеров

**Параметры:**
```python
lgb.LGBMRanker(
    objective='lambdarank',
    metric='ndcg',
    ndcg_eval_at=[4, 10],
    n_estimators=500,
    learning_rate=0.05,
    num_leaves=63,
    max_depth=6,
    min_child_samples=5,
    subsample=0.8,
    colsample_bytree=0.8,
)
```

**Inference:** `model.predict(X)` вместо `model.predict_proba(X)[:, 1]`

**CV результат (30 дней):**

| Метрика | Значение |
|---|---|
| overall_target_metric | **0.6661** |
| evaluated_days | 22 |
| evaluated_shifts | 137 |

## 5. Вывод: почему Classifier лучше Ranker на этих данных

### 5.1 Размер групп в Ranker

Обучение Ranker требует per-shift групп: позитивы + VIEW-only негативы на той же смене. При CV-разбиении (только 232K событий вместо 383K) многие группы маленькие — 1-2 позитива и 2-5 негативов. Lambdarank на малых группах нестабилен.

### 5.2 Отсутствие early stopping у Ranker

Classifier использует AUC-based early stopping на held-out 15%, что ограничивает переобучение. Ranker обучается фиксированные 500 деревьев без валидации — потенциальное переобучение.

### 5.3 Диапазон негативных примеров

- **Classifier:** глобальные VIEW-only негативы — разнообразные, покрывают всё пространство пользователей
- **Ranker:** только VIEW-пользователи той же смены — узкое распределение, меньше обобщения

### 5.4 Дни, которые Ranker "видит", но Classifier не оценивает

Ranker оценивает Feb 1, 4, 5 с метриками 0.17, 0.17, 0.44. Это дни, где Ranker сумел ранжировать хотя бы одного позитива в top-K, но с очень низким качеством. Classifier на тех же сменах не попал ни одним позитивом в top-K. Это компромисс: Ranker лучше в "recall" (находит больше оцениваемых ситуаций), но хуже в "precision" (качество ранжирования ниже).

---

## 6. Решение: оставить LGBMClassifier

**LGBMRanker убирается из основного пути.** Classifier+AUC остаётся как продакшн-модель.

Причины:
- CV показывает +9.5 пп разрыва в пользу Classifier на реальных данных (0.761 vs 0.666)
- На официальном eval оба дают 0.750 (потолок данных), разницы нет
- Classifier более стабилен: 12/19 дней лучше, Ranker лучше только в 4/19

**Правило:** на малом датасете (< ~500K обучающих пар per shift) классификатор с качественными глобальными негативами надёжнее ранжировщика.

## 8. Ограничения CV и что можно улучшить

### 8.1 Размер обучения при CV меньше, чем в продакшне

При 30-дневном CV обучение идёт на 232K событий вместо 383K в продакшне. Это занижает CV-метрику относительно реального теста — продакшн-модель сильнее.

### 8.2 Состав val периода

Val-окно (янв 15 – фев 14) предшествует реальному тесту (фев 17+). Возможен дрейф данных: состав смен, локаций и работников мог измениться.

### 8.3 Рекомендация

Для более надёжной оценки лучше использовать множественные разбиения:
- 14-дневное окно: слишком мало данных (2 оцениваемых дня)
- 30-дневное окно: оптимально (22 дня, 121–137 смен)
- Оба разбиения дают одинаковую рекомендацию по модели
