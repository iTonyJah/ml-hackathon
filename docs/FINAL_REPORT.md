# Итоговый отчет по проекту ранжирования кандидатов на смены


## Кратко

В проекте решалась задача: по описанию смены вернуть список работников, которые с наибольшей вероятностью подадут заявку и выйдут на эту смену.

Изначально сервис был baseline-решением: он выбирал пользователей в основном по локации и возвращал их почти без умного ранжирования. В финальной версии добавлена ML-модель на LightGBM, кэши для быстрого inference, отдельная логика для "спящих" работников и time-based cross-validation для более честной проверки качества.

Финальная логика работает так:

1. Сервис получает пользователей, смены и события через RPC.
2. Метод `prepare` загружает данные из SQLite, обучает модель и строит кэши в памяти.
3. Метод `predict` берет кандидатов из кэша, считает ML-score и возвращает top-N пользователей.
4. После ML-скоринга применяется дополнительная перестановка: пользователи с историей по этой смене и "спящие" надежные работники получают приоритет.

## Задача простыми словами

Есть платформа временной занятости. Работодатель публикует смену: место, время, тип работы, оплата, длительность и сколько человек нужно. Пользователи смотрят смены, подают заявки, иногда отменяют заявки, иногда выходят и успешно отрабатывают.

Нужно сделать сервис, который по новой смене быстро отвечает: "Вот список людей, которым эту смену стоит показать первыми".

Это не просто бинарная классификация "подойдет / не подойдет". На практике важен порядок. Если сервис вернул 10 человек, самые перспективные должны быть наверху.

## Данные

В проекте используются три основные таблицы:

- `user.csv` - данные о пользователях;
- `shift.csv` - данные о сменах;
- `event.csv` - история взаимодействий пользователей со сменами.

Основные типы событий:

| Событие | Что означает |
|---|---|
| `VIEW` | пользователь посмотрел смену |
| `APPLY` | пользователь подал заявку |
| `FINISHED` | пользователь вышел и отработал |
| `USER_CANCEL` | пользователь отменил заявку |
| `SYSTEM_CANCEL` | смена отменена системой |

Для оценки используется validation-набор:

- `validation/shift.csv` - смены, по которым нужно сделать рекомендации;
- `validation/event.csv` - события, которые догружаются день за днем;
- `validation/apply.csv` - правильные ответы: кто реально подал заявку на какую смену.

Важный момент: в `apply.csv` колонка `date` должна быть датой начала смены, то есть `shift.start_at.date()`, а не датой события `APPLY`.

## Как считается качество

Eval вызывает `predict` для validation-смен и смотрит, попал ли нужный пользователь в верхнюю часть списка.

Основные правила:

- фиксированный пул для метрики - TOP-10 кандидатов;
- для смены с `capacity = K` оценивается верхняя часть списка с учетом этой емкости;
- используется ROC-AUC с ограничением FPR:

```text
max_fpr = min(1.0, capacity / 10)
```

Метрика агрегируется в несколько шагов:

1. Сначала по отдельным сменам.
2. Затем внутри групп с одинаковым `capacity`.
3. Затем по дням.
4. Потом среднее по всем дням.

У этой метрики есть важная особенность: смены с `capacity = 1` часто не дают оцениваемой ROC-AUC ситуации, потому что в одном объекте нельзя получить оба класса сразу. Поэтому при анализе результата всегда нужно смотреть не только итоговое число, но и число реально оцениваемых смен и групп.

## Что было в baseline

В исходной версии сервис работал очень просто:

- пользователи, смены и события записывались в SQLite;
- `prepare` почти ничего не делал;
- `predict` искал кандидатов SQL-запросом;
- ранжирование было слабым и почти не использовало историю пользователя.

Такой подход работал как техническая заглушка, но плохо использовал данные:

- не учитывал тип задачи;
- почти не учитывал историю пользователя;
- не понимал, работал ли пользователь у этого работодателя раньше;
- не учитывал повторяющиеся смены;
- ходил в базу в момент `predict`, что плохо для latency.

## Финальное решение

Финальное решение состоит из четырех основных частей.

### 1. Кэш кандидатов

В `prepare` строится список пользователей по локациям. Для каждой локации пользователи отсортированы по активности. Это дает быстрый способ получить разумный пул кандидатов для смены.

В `predict` берется до 300 кандидатов:

- для крупных локаций - пользователи из этой локации;
- для маленьких локаций - пользователи из локации плюс fallback из глобального топа активных пользователей.

Ограничение до 300 кандидатов оказалось полезным: слишком большой пул добавляет шум и может вытеснять релевантных людей.

### 2. ML-модель

Основная модель - `LGBMClassifier`.

Она учится отвечать на вопрос: "Какова вероятность, что этот пользователь подаст заявку на эту смену?"

Позитивные примеры:

- пары `user_id + shift_id`, где есть `APPLY` до начала смены;
- смены с `SYSTEM_CANCEL` исключаются.

Негативные примеры:

- пары, где был `VIEW` или `USER_CANCEL`, но не было последующего `APPLY`.

Для борьбы с дисбалансом берется ограниченное число негативов относительно позитивов.

### 3. Признаки

Модель использует признаки нескольких типов.

История пользователя:

- доля заявок;
- доля успешных выходов;
- доля отмен;
- число заявок;
- число просмотров;
- число активных дней;
- наличие медкнижки.

Параметры смены:

- час начала;
- день недели;
- длительность;
- оплата;
- емкость;
- требуется ли медкнижка;
- тип ставки.

Совместимость пользователя и смены:

- совпадает ли локация;
- подходит ли пользователь по медкнижке;
- работал ли он с этим работодателем;
- работал ли он на этой точке;
- насколько оплата похожа на его прошлые успешные смены.

История по конкретной смене:

- подавал ли пользователь заявку на эту смену раньше;
- выходил ли на нее;
- отменял ли;
- как давно была последняя заявка.

### 4. Sleeper-rerank

В данных обнаружилась важная группа пользователей: у них нет `APPLY` в обучающем окне, но есть `FINISHED`.

Это значит, что пользователь реально выходил на смены, но его заявка могла быть сделана до начала доступной истории. Для обычной модели такой человек выглядит слабым кандидатом: заявок нет, apply-rate равен нулю. Но фактически это надежный работник.

Таких пользователей назвали "спящими".

После ML-скоринга применяется трехуровневый rerank:

1. Сначала пользователи, которые уже подавали заявку на эту конкретную смену.
2. Затем "спящие" пользователи: нет `APPLY`, но есть `FINISHED`.
3. Затем все остальные по ML-score.

Это не заменяет модель, а добавляет поверх нее бизнес-сигнал, который модель плохо видит из-за границ обучающего окна.

## Как работает `prepare`

`prepare` - тяжелая часть пайплайна. Она выполняется не на каждый запрос, а отдельно:

1. Загружает `users`, `shifts`, `events` из SQLite.
2. Приводит даты к нужным типам.
3. Строит статистики пользователей и работодателей.
4. Формирует обучающие пары.
5. Обучает `LGBMClassifier` с AUC early stopping.
6. Строит numpy-кэш для быстрого inference.
7. Строит кэш кандидатов по локациям.
8. Переводит сервис в состояние `ready`.

Это сделано специально: все тяжелые операции уходят в подготовку, а `predict` остается быстрым.

## Как работает `predict`

`predict` - горячий путь сервиса. Он должен отвечать быстро.

Последовательность такая:

1. Проверить, готов ли сервис.
2. Получить кандидатов из in-memory кэша.
3. Если кэш еще пуст, использовать fallback через БД.
4. Собрать признаки для кандидатов и текущей смены.
5. Получить ML-score.
6. Применить sleeper-rerank и recency-rerank.
7. Вернуть первые `limit` пользователей.

В нормальном режиме `predict` не делает тяжелых SQL-запросов.

## Эксперименты

В ходе работы наша команда сравнивала разные подходы:

- baseline SQL-ранжирование;
- LightGBM-классификатор;
- изменение размера пула кандидатов;
- sleeper-rerank;
- AUC early stopping;
- LGBMRanker / LambdaRank;
- time-based cross-validation.

Главный вывод: на этих данных `LGBMClassifier` оказался стабильнее `LGBMRanker`.

Причины:

- группы для ranking-модели часто маленькие;
- у классификатора шире и стабильнее негативные примеры;
- AUC early stopping помогает ограничить переобучение;
- официальный validation слишком мал для надежного сравнения моделей, поэтому важнее смотреть time-based CV.

## Результаты

Зафиксированные результаты из рабочих отчетов:

| Контекст | Результат | Комментарий |
|---|---:|---|
| Стартовое baseline-решение | 0.534 | слабое SQL-ранжирование |
| Локальный validation после ML и sleeper-rerank | 0.750 | ограничение конкретного val-набора |
| После исправления split и target construction | 0.800 | корректнее собран ground truth |
| Time-based CV, `LGBMClassifier` | 0.7612 | 121 оцениваемая смена |
| Time-based CV, `LGBMRanker` | 0.6661 | хуже и нестабильнее |
| Новый увеличенный датасет | 0.978 | больше истории, лучше качество |

Эти числа нельзя смешивать как одну линию leaderboard. Они относятся к разным validation-контекстам. Поэтому в финальной документации важно всегда указывать, на каком наборе данных и каким способом получена метрика.

## Ограничения

У решения есть ограничения:

- качество сильно зависит от качества исторических событий;
- если у пользователя совсем нет истории, модель опирается на fallback-признаки;
- локальный validation может быть слишком мал для надежного сравнения моделей;
- `capacity=1` может создавать дни, которые почти не влияют на ROC-AUC;
- sleeper-rerank полезен для найденного паттерна, но это эвристика, ее нужно проверять на новых данных.

## Что можно улучшить дальше

Возможные направления:

- сделать несколько time-based CV окон вместо одного;
- добавить больше признаков по свежести активности пользователя;
- аккуратнее учитывать отмены;
- отдельно оценивать cold-start пользователей и смены;
- добавить мониторинг качества candidate pool;
- сохранять модель и кэши как артефакты, если потребуется быстрый рестарт сервиса;
- сделать подробный отчет по feature importance.

## Итог

Финальное решение превратило baseline-сервис в полноценный рекомендательный сервис для смен:

- данные загружаются через существующий RPC-контракт;
- модель обучается в `prepare`;
- быстрый inference работает через in-memory кэши;
- ранжирование учитывает историю пользователя, параметры смены и специальные паттерны данных;
- качество проверяется не только официальным eval, но и более надежным time-based CV.

Главная инженерная идея: не пытаться делать все в `predict`. Все тяжелое готовится заранее, а в момент запроса сервис только быстро достает кандидатов, скорит их и возвращает top-N.
