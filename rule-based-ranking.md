# Rule-based ranking

Документ описывает rule-based ранжирование кандидатов, примененное в проекте. Основная реализация находится в `ml-hackathon/hackaton/service/repositories.py`.

## Краткая идея

Сервис должен вернуть список `user_ids` для конкретной смены. Rule-based часть решает две задачи:

1. Сформировать ограниченный пул кандидатов, которые потенциально подходят под смену.
2. Отсортировать этих кандидатов по эвристическому `rule_score`, рассчитанному из профиля пользователя, параметров смены и исторических взаимодействий.

Rule-based ranking является базовым ранжированием. В текущей версии проекта поверх него может применяться ML reranker, если включен `ENABLE_ML_RERANKER`. Даже при включенном reranker `rule_score` остается важным сигналом: он входит в blend-формулу и используется как tie-breaker.

## Где используется в online inference

В `predict` сервис делает следующие шаги:

1. Проверяет, что `prepare` завершен.
2. Валидирует входной payload и извлекает `shift`.
3. Вызывает `Repository.find_scored_candidates(shift, limit=candidate_pool_limit)`.
4. Обрезает результат до запрошенного `request.limit`.
5. Если scored-кандидатов нет, использует fallback по локации и медкнижке.
6. Если и он пустой, возвращает первых пользователей по `id`.

Размер первичного пула задается как:

```text
candidate_pool_limit = max(request.limit, CANDIDATE_POOL_LIMIT)
```

По умолчанию `CANDIDATE_POOL_LIMIT = 1000`, но не меньше `10`.

## Prepare-time агрегаты

Перед ранжированием сервис строит агрегаты по истории событий. Это делается в `Repository.rebuild_features()`.

Создаются четыре группы признаков:

1. `user_features`: общая история пользователя по всем сменам.
2. `user_task_features`: история пользователя по конкретному `task_type`.
3. `user_employer_features`: история пользователя по конкретному `employer_id`.
4. `user_workplace_features`: история пользователя по конкретному `workplace_id`.

Для этих групп считаются:

- `view_cnt`: сколько раз пользователь видел смены.
- `apply_cnt`: сколько раз пользователь откликался.
- `finished_cnt`: сколько раз пользователь завершал смены.
- `user_cancel_cnt`: сколько раз пользователь отменял.
- `system_cancel_cnt`: сколько раз была системная отмена.
- `avg_reward_per_hour`: средняя ставка пользователя за час по событиям `APPLY` и `FINISHED`.

Разделение агрегатов по `task_type`, `employer_id` и `workplace_id` нужно, чтобы сильнее поднимать кандидатов с релевантной историей именно для похожей смены, а не только с хорошей общей активностью.

## Candidate generation

Основной SQL для online-кандидатов находится в `_find_candidate_feature_rows()`.

Кандидат должен пройти жесткий фильтр по медкнижке:

```sql
(shift.need_mk = 0 OR user.has_mk = 1)
```

То есть если смена требует медкнижку, пользователь без медкнижки не попадает в пул.

После этого пользователь попадает в пул, если выполняется хотя бы одно условие:

- `user.location_id = shift.location_id`: пользователь из той же локации.
- `user.is_strict_location = 0`: пользователь не привязан строго к одной локации.
- есть история пользователя по этому `task_type`.
- есть история пользователя по этому `employer_id`.
- есть история пользователя по этому `workplace_id`.

Такой фильтр расширяет пул за пределы точного совпадения локации, но только для пользователей с признаками переносимости или релевантной историей.

## Состав `rule_score`

`rule_score` является суммой положительных и отрицательных эвристических вкладов. Чем выше score, тем выше кандидат в rule-based выдаче.

Формула в online inference:

```text
rule_score =
    location_match_bonus
  + mk_bonus
  + strict_location_match_bonus
  + global_history_score
  + task_history_score
  + employer_history_score
  + workplace_history_score
  + conversion_rate_score
  - view_without_apply_penalty
  - reward_mismatch_penalty
```

### Совпадение с параметрами смены

Базовые бонусы:

| Сигнал | Условие | Вклад |
|---|---:|---:|
| Совпадение локации | `user.location_id = shift.location_id` | `+30.0` |
| Пользователь имеет медкнижку | `user.has_mk = 1` | `+8.0` |
| Строгая локация совпала | `user.is_strict_location = 1` и локация совпала | `+5.0` |

Важно: бонус за медкнижку начисляется всем пользователям с `has_mk = 1`, даже если смена не требует медкнижку. Если смена требует медкнижку, пользователи без нее отфильтровываются раньше.

### Общая история пользователя

Общая история берется из `user_features`.

| Сигнал | Вклад |
|---|---:|
| Завершенная смена | `finished_cnt * 3.0` |
| Отклик | `apply_cnt * 2.0` |
| Просмотр | `view_cnt * 0.15` |
| Отмена пользователем | `-user_cancel_cnt * 1.5` |
| Системная отмена | `-system_cancel_cnt * 0.5` |

Интерпретация:

- `FINISHED` ценится выше `APPLY`, потому что подтверждает реальный выход на смену.
- `VIEW` дает небольшой плюс как слабый сигнал интереса.
- отмены штрафуются, причем пользовательская отмена сильнее системной.

### История по типу задачи

Агрегаты берутся только для `shift.task_type`.

| Сигнал | Вклад |
|---|---:|
| Завершение похожей задачи | `task_finished_cnt * 12.0` |
| Отклик на похожую задачу | `task_apply_cnt * 7.0` |
| Просмотр похожей задачи | `task_view_cnt * 0.5` |

Этот блок весит сильнее общей истории, потому что совпадение по типу задачи ближе к текущей смене.

### История по работодателю

Агрегаты берутся только для `shift.employer_id`.

| Сигнал | Вклад |
|---|---:|
| Завершение смены у этого работодателя | `employer_finished_cnt * 18.0` |
| Отклик на смену этого работодателя | `employer_apply_cnt * 9.0` |
| Просмотр смены этого работодателя | `employer_view_cnt * 0.5` |

История по работодателю весит выше истории по типу задачи: повторная успешная работа с тем же работодателем считается сильным сигналом.

### История по рабочему месту

Агрегаты берутся только для `shift.workplace_id`.

| Сигнал | Вклад |
|---|---:|
| Завершение смены на этом рабочем месте | `workplace_finished_cnt * 24.0` |
| Отклик на смену на этом рабочем месте | `workplace_apply_cnt * 12.0` |
| Просмотр смены на этом рабочем месте | `workplace_view_cnt * 0.5` |

Это самый сильный исторический блок. Логика такая: если пользователь уже выходил на конкретное рабочее место, вероятность релевантности выше, чем просто совпадение по типу задачи или работодателю.

## Conversion-rate сигналы

Помимо абсолютных счетчиков, формула учитывает долю откликов относительно просмотров и откликов.

Общая формула:

```text
apply_rate = apply_cnt / max(1, view_cnt + apply_cnt)
```

Веса:

| Уровень | Формула | Максимальный вклад |
|---|---|---:|
| Пользователь в целом | `user_apply_rate * 5.0` | `+5.0` |
| Тип задачи | `task_apply_rate * 15.0` | `+15.0` |
| Работодатель | `employer_apply_rate * 18.0` | `+18.0` |
| Рабочее место | `workplace_apply_rate * 20.0` | `+20.0` |

Эти признаки помогают отличать пользователя, который много смотрит, но редко откликается, от пользователя с высокой конверсией в отклик.

## Штраф за просмотры без отклика

Для `task_type`, `employer_id` и `workplace_id` добавлен штраф за ситуацию, когда пользователь много смотрел похожие смены, но не откликался.

Формула:

```text
view_without_apply_rate =
    max(0, view_cnt - apply_cnt) / max(1, view_cnt + apply_cnt)
```

Штрафы:

| Уровень | Формула | Максимальный штраф |
|---|---|---:|
| Тип задачи | `task_view_without_apply_rate * 4.0` | `-4.0` |
| Работодатель | `employer_view_without_apply_rate * 5.0` | `-5.0` |
| Рабочее место | `workplace_view_without_apply_rate * 6.0` | `-6.0` |

Общий пользовательский `view_without_apply_rate` считается как ML-признак, но в текущую online rule-based формулу напрямую не входит.

## Штраф за несоответствие ставки

Для пользователя считается `avg_reward_per_hour`: средняя ставка за час по сменам, где были `APPLY` или `FINISHED`.

Для текущей смены считается:

```text
reward_per_hour = shift.reward / shift.hours
```

Если у пользователя есть историческая средняя ставка, начисляется штраф:

```text
reward_mismatch_penalty =
    min(20.0, abs(avg_reward_per_hour - reward_per_hour)) * 0.05
```

Максимальный штраф равен `1.0`. Он небольшой относительно исторических бонусов, поэтому ставка используется как корректирующий сигнал, а не как основной фактор ранжирования.

## Итоговая сортировка без ML reranker

SQL сразу сортирует кандидатов так:

```sql
ORDER BY
    rule_score DESC,
    u.location_id = shift.location_id DESC,
    u.has_mk DESC,
    u.id ASC
LIMIT candidate_pool_limit
```

То есть при равном `rule_score` выше будут:

1. пользователи из той же локации;
2. пользователи с медкнижкой;
3. пользователи с меньшим `id` в лексикографическом порядке.

Если ML reranker выключен или не обучен, именно этот порядок возвращается из `find_scored_candidates()`.

## Связь с ML reranker

Rule-based ranking не удален после добавления ML. Он используется в двух местах:

1. Для первичной сортировки и ограничения candidate pool.
2. Как часть финального score при включенном ML reranker.

Когда reranker готов, для каждого кандидата считается:

```text
normalized_rule_score =
    (rule_score - min_rule_score) / max(max_rule_score - min_rule_score, 1.0)

ml_score =
    (1 - ML_RERANKER_WEIGHT) * normalized_rule_score
  + ML_RERANKER_WEIGHT * model_score
```

По умолчанию `ML_RERANKER_WEIGHT = 0.5`.

После этого кандидаты сортируются по:

1. `ml_score DESC`;
2. `rule_score DESC`;
3. `location_match DESC`;
4. `has_mk DESC`;
5. `user_id ASC`.

Таким образом rule-based score остается страховочным и стабилизирующим сигналом даже в гибридном режиме.

## Fallback ranking

Если основной scored-пул пустой, используется `find_top_candidates()`:

```sql
SELECT id
FROM users
WHERE location_id = shift.location_id
  AND (shift.need_mk = 0 OR has_mk = 1)
ORDER BY is_strict_location DESC, has_mk DESC, id ASC
LIMIT request.limit
```

Этот fallback уже не использует историю событий. Он выбирает пользователей из нужной локации, соблюдает требование медкнижки и сортирует по:

1. строгому совпадению локации;
2. наличию медкнижки;
3. `id`.

Если и такой список пустой, применяется последний fallback:

```sql
SELECT id FROM users ORDER BY id ASC LIMIT request.limit
```

Он нужен только чтобы сервис мог вернуть ответ при наличии пользователей, даже если нет подходящих кандидатов по правилам.

## Почему веса устроены именно так

Формула отражает несколько практических предположений:

- совпадение локации важно, поэтому получает большой базовый бонус;
- успешная история важнее простого просмотра;
- релевантная история по рабочему месту сильнее истории по работодателю, а история по работодателю сильнее истории по типу задачи;
- отклик без завершения полезен, но слабее `FINISHED`;
- просмотры без отклика могут означать низкий интерес, поэтому штрафуются;
- ставка за час учитывается мягко, потому что слишком сильный штраф мог бы выбрасывать пользователей с хорошей релевантной историей.

## Ограничения подхода

Rule-based ranking прозрачен и быстрый, но имеет ограничения:

- веса заданы вручную и не оптимизируются автоматически;
- формула линейная и не ловит сложные взаимодействия признаков;
- пользователи без истории получают мало сигналов, кроме локации и медкнижки;
- большой исторический счетчик может доминировать над свежестью событий, потому что в текущей формуле нет decay по времени;
- `id ASC` как последний tie-breaker стабилен, но не несет бизнес-смысла.

Именно поэтому в проекте добавлен ML reranker: он использует те же подготовленные признаки и `rule_score`, но может донастроить порядок внутри candidate pool по данным.
